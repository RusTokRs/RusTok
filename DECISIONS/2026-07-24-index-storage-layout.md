# ADR: Select the physical PostgreSQL layout for `rustok-index`

- Date: 2026-07-24
- Status: Proposed
- Owners: RusToK platform / Index module
- Scope: PostgreSQL entity/link persistence for the generic Index Engine

## Context

The Index Engine requires one production PostgreSQL representation for generic,
tenant- and locale-scoped entities plus independently stored cross-module links.
Production migrations are intentionally blocked until the M2 benchmark compares
the three implemented candidates at both `100k` and `1m` Product-locale rows.

The candidates are:

1. JSONB entity rows with typed expression and GIN indexes;
2. normalized typed EAV field-value rows;
3. specialized hot typed projections as a best-case comparison baseline.

The benchmark contract, workloads, invariants, and evidence fields are defined in
`crates/rustok-index/docs/storage-benchmark.md`. Cross-scale summaries are
generated by `scripts/verify/compare-index-storage-evidence.mjs` according to
`crates/rustok-index/docs/storage-comparison.md`.

The archived smoke packet proves harness sanity only and cannot select a model.

## Evidence required before acceptance

- an inspected `100k` read, mutation/WAL, and maintenance packet;
- an inspected `1m` read, mutation/WAL, and maintenance packet;
- exact entity/link cardinality and read-result parity across candidates;
- first-run and warm-run latency/buffer comparison for every read workload;
- planner-shape stability at both scales;
- load duration and relation-size scaling;
- update/delete latency and WAL amplification;
- committed churn, dead-tuple estimates, HOT updates, VACUUM duration, and
  pre/post-VACUUM size behavior;
- an operational-complexity review for dynamic fields, schema evolution,
  secondary indexes, migrations, and operator diagnostics.

## Inspected 100k evidence

Actions run `30051321255` and artifact
`index-storage-100k-84a11b147689b226ca161f5a0287990c1e8489d4`
passed the scale validator for PostgreSQL 16, three repetitions, and five churn
cycles. All candidates preserved 300,080 entities and 600,000 links, returned
identical read rows/digests, matched mutation effects, and preserved cardinality
through churn and VACUUM.

The 100k packet establishes these provisional findings:

- the hot typed projection is the size/load/read/write best-case baseline, but it
  does not satisfy the generic dynamic-schema requirement by itself;
- JSONB is materially smaller and faster than typed EAV across this packet;
- typed EAV has the largest relation size, slowest load, highest delete WAL, and
  highest post-churn dead-tuple estimate;
- the pre-fix two-hop workload was unacceptable for all three candidates;
  EXPLAIN identified missing typed-link target predicates, which are now fixed and
  require a same-commit 100k/1m rerun before comparison;
- all plan shapes are stable across the three repetitions at 100k;
- ordinary VACUUM clears dead-tuple estimates but does not shrink relation files.

The inspected run's `1m` stage did not run because repository variable
`INDEX_BENCH_LARGE_RUNNER` was unset. The 100k runner nevertheless reported
93,030,404,096 free root-filesystem bytes before evidence and 88,893,792,256 after.
The workflow now uses the configured larger-runner label when present and falls
back to `ubuntu-latest` otherwise; the reusable job remains fail-closed below
35,000,000,000 free bytes. The 1m packet is still pending, so these findings are
not sufficient to accept the ADR.

## Decision

Pending scale evidence. No production model is selected by this proposed ADR.

When accepted, this section must name exactly one physical model, identify any
required changes from the benchmark prototype, and state the production indexing
and partitioning rules.

## Rejected alternatives

Pending. The accepted revision must record a specific evidence-backed rejection
reason for every unselected candidate. The hot projection remains a comparison
baseline unless the generic-engine constraints are changed by a separate ADR.

## Consequences

While Proposed:

- production Index migrations remain intentionally absent;
- M3 storage-engine implementation must not start;
- all three benchmark candidates remain available for evidence generation;
- smoke, `100k`, and `1m` results must not be mixed across incompatible commits or
  PostgreSQL configurations.

After acceptance:

- implement only the selected physical model in production migrations;
- delete rejected benchmark prototypes after preserving the archived evidence;
- update the implementation plan and storage benchmark documentation;
- add PostgreSQL integration tests for tenant/locale isolation, source-version
  guards, atomic entity/link mutations, rollback, redelivery, and concurrency.
