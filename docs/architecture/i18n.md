# i18n Architecture

This document captures the i18n contract of the current state in RusToK: where the
effective locale is selected, how the storage contract is structured and how UI should consume language.

## Basic Principle

i18n in RusToK is not a local convention of an individual module or host.
It is a platform contract.

Consequences:

- effective locale is selected by the server/runtime layer
- storage and transport use a single locale contract
- module-owned UI packages do not introduce their own locale fallback chain
- localized data lives in parallel records, not chaotically in base rows

## Effective Locale

Effective locale is selected by the host/runtime layer based on:

- tenant locale policy
- request context
- server-side normalization and validation

The UI package receives the already computed effective locale and must not select it
through a query/header/cookie fallback chain.

## Storage Contract

The current target storage pattern:

- language-agnostic state lives in base tables
- localized short fields live in `*_translations`
- heavy localized content may live in `*_bodies`
- locale is stored in normalized form with a safe width of `VARCHAR(32)`

Product catalog attributes follow the same pattern: attribute labels,
option labels, category metadata and schema/group labels live in translation
tables; text-like product attribute values may be localized, while number,
boolean, date and option values remain locale-neutral.

Inline localized JSON or mixed storage is not the target pattern for new module
work; move localized fields into owner translation tables when touching that
surface.

The machine-readable DB registry is
`database-multilingual-contract.json`. It declares only concretely guarded
surfaces as enforced and keeps legacy mixed-storage or narrow-locale schemas as
explicit owner-owned gaps. The human-readable audit is
`database-multilingual-audit.md`.

## UI Contract

A single rule applies to UI:

- Leptos hosts use host-provided locale context
- Next hosts use runtime/host locale providers
- module-owned UI packages only consume the effective locale
- package-local locale negotiation is prohibited
- shared support crates for module-owned UI also follow this rule:
  they do not introduce editable locale fields and do not invent their own fallback chain on top of host locale

This rule is equally important for admin, storefront and capability-owned UI surfaces.

## Transport Contract

i18n must be consistent across all layers:

- request locale selection
- GraphQL and REST payload semantics
- module-owned UI bundles
- email/template rendering
- search/index/read-model projections, if locale affects the read side

The transport layer must not have its own separate locale contract that diverges from
runtime and storage.

## Verification

Minimum i18n gates:

- `npm run verify:i18n:ui`
- `npm run verify:i18n:contract`
- `node scripts/verify/verify-db-multilingual-contract.mjs` for the focused DB contract
- host/module-checks, if locale-aware routing or bundle wiring changes

`verify:i18n:contract` includes the DB multilingual verifier, so existing CI
consumers receive the language-agnostic storage checks without a separate npm
entrypoint.

If a module's i18n contract changes, its local docs must be updated as well.

## Current Runtime Status

- Next hosts take supported locales from host-owned message loaders and do not hold
  a separate package-local locale registry.
- `apps/next-frontend` uses a common locale-aware middleware matcher without
  a hardcoded `/en|/ru` route filter.
- `apps/next-admin` selects the effective locale at the host/runtime level:
  query `locale`, cookie `rustok-admin-locale` and request headers are normalized into `x-rustok-effective-locale`
  in the order `?locale` → cookie → header → `Accept-Language` → `en`,
  while UI and module-owned packages consume the result via `next-intl` / `<html lang>`.
- Leptos hosts use a generated locale enum from current locale bundles; adding
  a new language starts with host-owned bundle/generation, not with a module-local fallback chain.
- Server locale middleware limits the effective locale to tenant policy from `tenant_locales` through
  a bounded TTL cache; counter metrics `rustok_tenant_locale_cache_hits_total`,
  `rustok_tenant_locale_cache_misses_total`, `rustok_tenant_locale_db_queries_total`,
  `rustok_tenant_locale_cache_invalidations_total` and gauge
  `rustok_tenant_locale_cache_entries` are used to control hot-path DB amplification.

## What Not To Do

- do not introduce a module-local locale chain
- do not treat the UI bundle contract as a separate source of truth from the runtime locale policy
- do not store localized business text in base rows if the module already follows the
  parallel-localized-record path
- do not spread locale normalization across multiple inconsistent layers
